# MBclawZZ

MBclawZZ 的总工作区。所有施工、归档、参考、交接都以此目录为根。

- 1-母体端
- 2-后端
- 3-APK端
- 4-总览端
- 5-杂项端
- 6-交接工作区域

- 参考文件

## 施工原则
- 先盘点，再施工。
- 先设计，再编码。
- 参考与主线分离。
- 旧版本进入归档，不直接覆盖。
每次修改，仅仅只能修改需要修的部分。
第一章 总原则
施工规范的作用，是防止项目在未来扩展时走样、失控或越改越乱。所有后续模块都必须遵守这些规范。

1.1 新模块必须先设计后编码
任何新功能、新目录、新接口都应该先有文档和结构，再写实现。

1.2 接口先行
设计阶段先预留标准接口文件，即使暂不实现，也要先把接口位置和用途说明放好。

1.3 不跨层访问内部实现
跨模块通信必须通过标准接口，不允许随手 import 内部类直接操作。

1.4 先保主线，再做优化
如果冲突，优先保证 MBclaw 能启动、能运行、能恢复，再谈重构美观。

1.5 小步提交思想
每次施工尽量只处理一个主题，不要把多个主题混在一起。

第二章 接口预留规范
每个模块至少包含：

interfaces/
├── api.py
├── service.py
├── events.py
├── models.py
├── errors.py
└── README.md
2.1 api.py
对外 API，负责稳定调用入口。

2.2 service.py
内部业务接口，负责业务实现。

2.3 events.py
事件定义，负责模块通信约束。

2.4 models.py
数据模型，负责结构说明。

2.5 errors.py
错误定义，负责异常收敛。

2.6 README.md
必须说明接口用途、调用方、被调用方、输入、输出、时机、生命周期、兼容性、扩展点。

第三章 统一调用规范
统一调用链：

Caller → Interface(API) → Service → Runtime → Result

禁止跨模块直接访问内部实现，不允许跳过中间层直接碰底层逻辑。

第四章 事件规范
事件统一为：

Event {
  id
  source
  target
  type
  timestamp
  payload
  metadata
}
所有跨模块通信优先通过事件总线完成，避免散乱调用。

第五章 版本规范
每个接口都必须维护：Version、Deprecated、ChangeLog。后续升级必须考虑向后兼容。
