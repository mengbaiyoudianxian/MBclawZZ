# 路线图 v2

**版本**: v2
**日期**: 2026-06-21
**节奏**: 每个阶段必须出可演示的闭环，否则不进下一阶段。

**目标**：清场。

| 任务 | 产出 | 责任 |

| 冻结 Lite 新功能开发 | 仓库置 README 警告 | CTO |

---

## R1 — 收敛到 MVP（2-3 周）

**目标**：3000 行可演示版本。

| 任务 | 产出 |
|---|---|
| 新建 `mbclaw-lite/r1` 分支 | 干净起点 |
| 实现 8 张表 schema | DDL + Alembic 初版 |
| 7 个 service 重构 | 每个 < 300 行 |
| 8 个 router 收敛 | 总 < 1500 行 |
| 统一 `llm_client` | 业务零 provider 依赖 |
| 端到端演示脚本 | 见 MVP §2 DoD |
| 分层测试 | service 单测 + 路由集成测 + 端到端 |

**门槛（同 MVP DoD）**：
1. 演示脚本通过：3 次会话后第 4 次能复用关键事实。
2. 代码总量 < 3000 行。
3. 文档与代码 1:1 同步。

---

## R2 — 智能增强（按需启用，无固定排期）

每项要**真实信号触发**：

| 子项 | 触发条件 |
|---|---|
| 向量检索 | M4 命中率 < 70% |
| 树状分类 | sessions > 100 |
| Curator 归档 | entries > 500 |
| Agent Runtime（新写，不复用） | 用户明确提自动化需求 |
| 工具索引 | 工具数 > 10 |
| 多模型调度 | 同时配置 ≥ 3 个 provider |
| 反馈/画像 | R1 上线后 ≥ 4 周用户数据 |
| ActionMemory / SkillCard | Agent Runtime 上线后 |
| TaskQueue / 优先级 | 出现真实并发任务 |

**门槛**：每项独立 PR，独立验证脚本，不能搭车合并。

## 时间约束

- R0：1 周（强制）

- R3：长期演进

---

## 反模式（一律拒绝）

| 反模式 | 拒绝理由 |
|---|---|
| "这功能很有意思，先加上" | MVP 期不允许 |
| "代码已经写好了，不删可惜" | 沉没成本谬误，进 Memory 归档 |
| "未来会有用" | 进 Design 讨论，不进 Core |
| "测试以后补" | DoD 包含测试，不补不算完 |
| "演示用假数据" | 演示必须用真实管线 |
