# MBclaw 当前架构分析

## 1. 当前项目结构

### 1.1 Token Pool (05-Token池/token_pool)
**技术栈**: Python (FastAPI) + SQLite
**核心组件**:
- `main.py`: FastAPI 应用入口，端口 8100
- `config.py`: 配置管理，支持环境变量和 .env 文件
- `pool/registry.py`: Key 注册表，SQLite 持久化，管理所有 API Key
- `pool/caller.py`: 统一调用器，处理 OpenAI/Anthropic 差异，自动故障转移
- `pool/scheduler.py`: 智能调度器，综合评分选最优 Key
- `pool/circuit.py`: 熔断器，每个 alias 独立熔断状态
- `pool/metrics.py`: 实时指标聚合，内存滑动窗口
- `pool/health.py`: 后台健康检测，定期探活所有 Key
- `routes/proxy.py`: OpenAI 兼容代理 `/v1/chat/completions`
- `routes/keys.py`: Key 管理 CRUD
- `routes/stats.py`: 统计与监控
- `routes/heartbeat.py`: 用户设备心跳
- `routes/admin.py`: 管理面板 HTML

**数据模型**:
- `keys` 表: 存储 API Key 信息（alias, provider, base_url, api_key, model, cost_per_1k, priority, enabled）
- `key_stats` 表: 存储 Key 运行时统计（status, success_count, fail_count, total_tokens, total_cost, avg_latency, last_checked, last_error）
- `call_log` 表: 存储调用日志（alias, ts, latency_ms, tokens, cost, success, error）

### 1.2 MiClaw Bridge (05-Token池/bridge_manager.py)
**技术栈**: Python (FastAPI) + HTTP 代理
**核心功能**:
- MiClaw 账号管理
- 登录流程（支持图形验证码和二步验证）
- 代理转发到 MiClaw API
- 实例生命周期管理

**关键端点**:
- `POST /bridge/miclaw/apply`: 创建白嫖实例
- `GET /bridge/miclaw/login/{id}`: 登录页面
- `POST /bridge/miclaw/login/{id}`: 验证 MiClaw 凭证
- `GET /bridge/miclaw/status`: 轮询实例状态
- `/bridge/miclaw/v1/*`: 代理 LLM 请求到 MiClaw

### 1.3 MiClaw Bridge Desktop (10-工具池/miclaw_bridge_src)
**技术栈**: Rust (Tauri) + TypeScript
**核心功能**:
- 桌面应用，提供本地代理服务
- 支持 OpenAI/Anthropic 代理
- Token 使用统计
- 系统托盘集成

**关键模块**:
- `src-tauri/src/proxy/openai.rs`: OpenAI 兼容代理，处理流式和非流式响应
- `src-tauri/src/usage.rs`: Token 使用统计，支持窗口化查询
- `src-tauri/src/auth/`: 认证模块（登录、刷新、二步验证）
- `src-tauri/src/server.rs`: HTTP 服务器

### 1.4 后端核心 (07-后端核心/app)
**技术栈**: Python (FastAPI/Flask)
**核心功能**:
- Agent 管理
- LLM 路由
- 工具注册
- 能力管理
- 管理面板

### 1.5 FreeLLMAPI (freellmapi)
**技术栈**: TypeScript (Express.js) + SQLite
**核心功能**:
- 免费 LLM API 聚合
- 多提供商支持（18 个提供商，161 个模型）
- 智能路由和故障转移
- 完整的统计和监控

**关键特性**:
- 自动更新模型目录
- 密钥加密存储
- 统一的 OpenAI 兼容接口
- 完整的 Web 管理界面

## 2. 调用流程

### 2.1 Token Pool 调用流程
```
客户端 → /v1/chat/completions → proxy.py
  ↓
调度器 pick_all_ranked(task) → 选择最优 Key 列表
  ↓
caller.call_with_fallback(payload, task)
  ↓
尝试第一个 Key:
  如果是 anthropic → _call_anthropic()
  否则 → _call_openai_compat()
  ↓
成功 → 记录指标，返回响应
失败 → 熔断器 on_failure()，尝试下一个 Key
  ↓
所有 Key 失败 → 返回 503 错误
```

### 2.2 MiClaw Bridge 调用流程
```
客户端 → /bridge/miclaw/apply → 创建实例
  ↓
用户访问登录页 → 输入小米账号密码
  ↓
POST /bridge/miclaw/login/{id} → 验证凭证
  ↓
轮询 /bridge/miclaw/status → 检查实例状态
  ↓
客户端 → /bridge/miclaw/v1/chat/completions
  ↓
代理转发到 MiClaw API (http://100.126.55.0:8765)
```

## 3. 依赖关系

### 3.1 Token Pool 依赖
- **外部依赖**: FastAPI, httpx, SQLite
- **内部依赖**: 无（独立服务）
- **配置依赖**: 环境变量 (TP_DATA, TP_HOST, TP_PORT, TP_ADMIN_KEY, TP_PROXY_KEY)

### 3.2 MiClaw Bridge 依赖
- **外部依赖**: FastAPI, httpx
- **内部依赖**: MiClaw API (http://100.126.55.0:8765)
- **配置依赖**: MICLAW_API_BASE, MICLAW_TOKEN_KEY

### 3.3 FreeLLMAPI 依赖
- **外部依赖**: Express.js, better-sqlite3, React (前端)
- **内部依赖**: 无（独立项目）
- **配置依赖**: .env 文件，环境变量

## 4. 存在问题

### 4.1 Token Pool 问题
1. **架构简单**: 单机部署，无水平扩展能力
2. **用户管理缺失**: 只有管理员，没有多用户支持
3. **配额管理缺失**: 无法限制每个用户的使用量
4. **模型路由简单**: 基于硬编码的任务类型路由
5. **统计功能有限**: 只有基本的调用日志，缺乏深度分析
6. **安全机制薄弱**: 管理接口使用简单的 API Key 验证
7. **MiClaw 集成粗糙**: 只是简单的代理转发，没有真正的账号管理

### 4.2 MiClaw Bridge 问题
1. **登录流程复杂**: 需要用户手动登录，无法自动维护会话
2. **实例管理简陋**: 基于文件存储，容易丢失
3. **错误处理不完善**: 连接失败时回退到备用模式
4. **无真正的多账号支持**: 只支持单个 MiClaw 账号
5. **Token 统计不准确**: 估算值，非实际使用量

### 4.3 架构问题
1. **项目分散**: Token Pool、MiClaw Bridge、FreeLLMAPI 是独立项目，缺乏统一架构
2. **技术栈不统一**: Python vs TypeScript vs Rust，维护成本高
3. **数据不共享**: 每个项目有自己的数据库，无法统一分析
4. **配置管理混乱**: 多个配置源，容易出错

## 5. 可以保留的部分

### 5.1 Token Pool 可保留
1. **熔断器设计**: `pool/circuit.py` 设计良好，可以复用
2. **指标收集**: `pool/metrics.py` 的滑动窗口设计值得保留
3. **健康检查**: `pool/health.py` 的定期检测机制可以保留
4. **API 设计**: RESTful 接口设计基本合理

### 5.2 MiClaw Bridge 可保留
1. **登录页面**: 前端登录页面设计可以保留
2. **代理转发**: 基本的代理转发逻辑可以保留
3. **实例生命周期管理**: 创建、销毁、暂停的概念可以保留

### 5.3 FreeLLMAPI 可借鉴
1. **多提供商支持**: 18 个提供商的集成方式值得借鉴
2. **智能路由**: 基于模型目录的路由逻辑值得借鉴
3. **密钥管理**: 加密存储和安全审查值得借鉴
4. **统计系统**: 完整的统计和监控系统值得借鉴

## 6. 应该重写的部分

### 6.1 用户系统
- 需要从零开始构建多用户系统
- 支持用户注册、登录、权限管理
- 支持 API Key 管理
- 支持使用配额和限制

### 6.2 配额管理
- 需要实现真正的配额管理系统
- 支持按用户、按模型、按时间窗口的限制
- 支持自动熔断和降级
- 支持配额共享（用户贡献 Key）

### 6.3 MiClaw 集成
- 需要重新设计 MiClaw 作为 Provider
- 支持多账号管理和轮询
- 自动维护登录状态和 Cookie
- 真正的 Token 统计，非估算

### 6.4 统计系统
- 需要构建统一的统计系统
- 支持多维度分析（用户、模型、时间、成本）
- 支持实时监控和告警
- 支持数据导出和报表

## 7. 应该删除的部分

### 7.1 Token Pool 删除
1. **硬编码的 Builtin Keys**: 应该通过配置或数据库管理
2. **简单的任务分类**: 应该基于模型能力动态路由
3. **文件存储的配置**: 应该使用数据库或配置中心

### 7.2 MiClaw Bridge 删除
1. **文件存储的实例管理**: 应该使用数据库
2. **硬编码的 API 地址**: 应该配置化
3. **简化的 Token 统计**: 应该使用真实的使用数据

### 7.3 架构层面删除
1. **独立的项目结构**: 应该统一到一个项目中
2. **重复的认证逻辑**: 应该统一认证系统
3. **分散的配置管理**: 应该统一配置管理
