# 对标矩阵

## 一、能力维度

| 维度 | 主要参考 | MBclaw 目标 | 结论 |
|---|---|---|---|
| 协议 / 接口 | MCP | 统一工具定义、接口稳定 | 参考协议形状，不引入重依赖 |
| 网关 / 渠道 | OpenClaw | QQ/微信/Web 统一入口 | 借鉴插件化与 session 绑定 |
| 记忆 | mem0 / GraphRAG | 长短期记忆分层 | 借鉴更新语义和未来知识图谱思路 |
| 模型路由 | LiteLLM / TokenPool | 统一模型入口、fallback、冷却 | 保留 TokenPool 为主，吸收冷却/重试逻辑 |
| Agent Runtime | OpenHands / Claude-Code / Codex-CLI | 任务拆解、执行、回滚、工具调用 | 借鉴 runtime 分层和 DI |
| 技能系统 | OpenClaw / agentskills | 工具与技能可发现可摘要 | 借鉴技能摘要与注册机制 |
| 视觉交互 | MBclaw 智能体之手方案 | 屏幕识别、点击、校准 | 保留自研思路，参考 UI grounding 做法 |
| 控制面板 | mbclaw-mother / OpenClaw 面板 | 总览、管理、健康检查 | 以现有主线为准，补交接与分区 |
| 桥接 / 适配 | miclaw_api_bridge / FreeLLMAPI | 平台接入与转发 | 仅保留桥接思想，不做多余二开 |

## 二、项目级对标结论

| 项目 | 可复用 | 可保留 | 可淘汰 |
|---|---|---|---|
| `mbclaw-mother` | 主线代码、gateway、memory、tokenpool 基础 | 主线运行流程、数据结构 | 重复实现、旧版本、备份文件 |
| `OpenClaw` | 渠道插件化、上下文引擎、skills 组织 | 渠道绑定、技能摘要 | 大规模 TS 全栈直接照搬 |
| `TokenPool` | 冷却、fallback、路由、健康检查 | key 管理、统计、调度 | 重复的 selector / 评分分叉 |
| `OpenHands` | runtime/SDK 分层、DI、integration 层 | 工具执行理念 | 沙箱/容器重依赖 |
| `LiteLLM` | cooldown、retry、rate limit、fallback | deployment 思路 | 复杂 router、预算、guardrails |
| `mem0` | 记忆增量更新、user_id 隔离 | 分类记忆 API | 外部向量 DB 强依赖 |
| `GraphRAG` | 层级摘要、关系记忆 | 未来知识层 | 现在直接引入图数据库 |
| `MCP` | Tool Definition、schema、发现机制 | 接口对齐 | 远程协议全量接入 |
| `Claude-Code` / `Codex-CLI` | 终端 agent flow、命令执行 | 任务推进思路 | 完整产品形态照搬 |
| `agentskills` | 技能说明组织、分层摘要 | skill 元数据 | 过多静态模板 |

## 三、研究优先级结论

### A. 必须保留研究
- `mbclaw-mother`
- `TokenPool`
- `OpenClaw`
- `OpenHands`

### B. 适合借鉴后收敛
- `LiteLLM`
- `mem0`
- `MCP`
- `Claude-Code`
- `Codex-CLI`
- `agentskills`

### C. 远期参考，不急于实现
- `GraphRAG`
- `FreeLLMAPI`
- `miclaw_api_bridge`

## 四、建议的对标方式

1. 先看职责边界，不看 UI 花样。
2. 先看数据流，不看目录名。
3. 先看可复用接口，再看实现细节。
4. 任何超过当前主线复杂度的能力，先标记为“远期”，不要直接引入。

## 五、结论

- **MBclaw 的主线不是把所有项目都照搬一遍**。
- **对标矩阵的目标是收敛，而不是扩张**。
- **最终可落地的只应保留：主线需要的边界、协议、记忆、调度、执行。**
