# 研究笔记

## 一、当前研究主线

### 1. mbclaw-mother
- 是当前最重要的主线仓库。
- 核心在于：Mother、Gateway、sub2api、Memory、Control Panel 之间的边界。
- 已有大量分析材料，说明它不是空壳，而是可运行体系。

### 2. OpenClaw
- 价值在于平台化和模块化程度高。
- 重点看：channels、context-engine、skills、plugin、registry。
- 对 MBclaw 的启发主要是“插件化”和“渠道/会话绑定”。

### 3. sub2api / LiteLLM
- sub2api 是 MBclaw 的模型入口核心。
- LiteLLM 提供冷却、fallback、retry、rate limit 的可借鉴思路。
- 研究目标不是复制企业级复杂网关，而是提炼可用策略。

### 4. OpenHands / Claude-Code / Codex-CLI
- 这三类更像“执行型智能体”的参考模板。
- 可借鉴 runtime 分层、DI、integrations、命令执行、任务收敛。
- 不要把沙箱、企业集成和大而全的能力直接搬进来。

### 5. mem0 / GraphRAG / MCP / agentskills
- 这些是记忆、知识、协议和技能组织的参考材料。
- 更适合做接口形状、数据结构、组织方式参考。

## 二、可复用结论

### 可复用
- Gateway 插件化思想。
- sub2api 的 cooldown / retry / fallback 思路。
- Memory 的增量更新与用户隔离思路。
- MCP 的工具 schema 和发现机制。
- OpenHands 的 runtime 分层与依赖注入。
- agentskills 的技能摘要组织。

### 可保留
- mbclaw-mother 的主线数据流。
- 当前已经确认可运行的服务和目录。
- 参考仓库本身，不急着改动。

### 可淘汰
- 重复实现的旧版文件。
- 备份/临时产物/重复 HTML。
- 过度设计的企业级模块。
- 与 MBclaw 主线无关的重型沙箱/预算/多租户功能。

## 三、当前判断

- 现阶段的关键不是“再多找项目”，而是“把已有项目边界收紧”。
- 重点要先把 Mother / sub2api / Gateway / Memory 的边界写清楚。
- 参考仓库只做“提炼”和“照镜子”，不做“全面重写”。

## 四、研究优先级建议

1. 先把 `mbclaw-mother` 的边界分析定稿。
2. 再把 `sub2api` 与 `LiteLLM` 的对照总结定稿。
3. 再看 `OpenClaw` 与 `OpenHands` 的架构分层。
4. 最后补 `mem0` / `GraphRAG` / `MCP` / `agentskills`。
