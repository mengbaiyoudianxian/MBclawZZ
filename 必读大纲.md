# 《MBclaw 操作系统（MBOS）总体设计规范》
MBclaw Overall Architecture Specification
Version：Planning v1.0

## 前言

MBclaw 不是聊天机器人，也不是一个单纯的问答壳，更不是把模型接口堆在一起就能成立的产品。MBclaw 的目标，是把“语言模型能力”提升为“长期运行的系统能力”，让 AI 从一次性对话工具，升级成一个可以持续工作、持续记忆、持续整理、持续协作、持续进化的 AI Operating System（AI 操作系统）。

这里的“操作系统”不是比喻而已。它意味着 MBclaw 要具备类似操作系统的组织方式：统一入口、统一调度、统一存储、统一上下文、统一工具、统一日志、统一配置、统一升级、统一归档。LLM 在这个系统里只是一名 Worker，不是系统本体；聊天只是系统的一个输入通道，不是系统边界；最终回复只是系统的输出之一，不是系统全部价值。

当前这份大纲不是愿景宣传，也不是产品文案，更不是抽象口号集合。它是 MBclawZZ 现在的施工总地基，是后续所有目录整理、仓库对标、文件归档、主线收缩、功能移植、接口预留、版本管理的最高级别文档。后续不管是盘点服务器资产、判断 GitHub 仓库用途、梳理 mbclaw-mother、整理 sub2api、研究 OpenClaw、比对 OpenHands，还是做归档、做总结、做交接，都必须先回到这份规范里确认方向，再继续动手。

这份文档要解决四个问题：
1. 先把 MBclaw 设计成什么样，说清楚。
2. 再把当前已经确认的资产、服务、仓库、目录说清楚。
3. 再把先做什么、后做什么、什么能留、什么能删、什么要归档，说清楚。
4. 最后把当前完成和未完成的事项摆出来，让后续 AI 不必每次重新翻找上下文。

MBclaw 当前采用的总分层，不是为了炫技，而是为了避免混乱：

基础设施 → 母体 → 客户端 → 平台 → 生态

这个顺序不是随便排的，而是按“最底层能力入口 → 中枢执行 → 用户触达 → 平台服务 → 长期扩展”来设计的。基础设施层负责统一资源入口、统一模型调度、统一配置与日志；母体层负责消息接入、上下文编排、记忆、思考、工具执行；客户端层负责 Android、Windows、Linux、Web 等落地；平台层负责官网、开发者中心、多语言、云同步与账户；生态层负责插件、市场、能力包与工作流。

这份大纲写得越明确，后续施工就越不容易偏航。它不是外宣稿，而是施工宪法。后续任何一次改目录、改模块、迁仓库、删文件、补文档，都必须先看这份规范是否允许。

## 第一篇：施工总原则（必须先看）

### 第一章 总原则

MBclaw 的设计和施工必须遵守以下原则，这些原则不是建议，而是约束。只要后续的实现与这些原则冲突，就必须优先调整实现，而不是默认忽略原则。

#### 1.1 单一职责（Single Responsibility）
每个模块只负责一类事情，不把多个层级的逻辑混在一起。比如：sub2api 只负责统一模型入口和调度，不负责 UI；Gateway 只负责消息接入和标准化，不负责推理；Memory 只负责记忆存储和检索，不负责上下文拼装；Runtime 只负责执行，不负责长期存储。职责一旦混乱，后续维护就会像泥巴一样越搅越糟。

#### 1.2 禁止重复实现（DRY）
任何功能都应尽量只保留一个“权威来源”。如果已经存在 sub2api 的调度逻辑，就不要在 Mother 里再写一套完全重复的逻辑；如果已经存在一份大纲，就不要在别的目录里用不同名字再抄一份；如果已经有对标矩阵，就不要在研究笔记里再重复写一遍同样结论。重复实现会让判断失真，最终导致“看起来很多，实际很乱”。

#### 1.3 Gateway 为唯一入口
所有平台入口都必须先经过 Gateway，再进入标准消息结构，再进入 Mother。无论是 QQ、微信、飞书、Web、CLI、Voice、Vision，还是未来其他平台，都不能绕开 Gateway 直达 Mother。Gateway 的存在不是为了增加一层，而是为了减少混乱。只要入口统一，后面就能统一处理身份、会话、权限、附件、回复格式。

#### 1.4 Mother 不感知平台
Mother 只处理标准化后的信息，不应该知道消息来自 QQ、微信还是 Web 页面。Mother 的输入应该是 StandardMessage，而不是某个平台的原始 payload。平台差异应该在 Gateway 里解决，母体只看统一格式。这样才不会在母体里堆出一堆平台 if/else。

#### 1.5 统一模型入口（sub2api）
模型入口层仍然是唯一模型入口；当前实现承载优先以 sub2api 为主，不再在 Mother 里并行维护第二套同类入口。任何模型调用原则上都应经过这层统一组织方式，再决定最终由哪个 provider、哪个 key、哪个模型、哪个 fallback 路径来承接。sub2api 负责 key 管理、provider 管理、调度、fallback、冷却、健康检测、统计；Mother 负责目标与策略判断，不重复吞掉整套入口逻辑。

#### 1.6 所有能力必须注册
技能、工具、模型、通道、适配器、命令、插件，都要能被注册、被发现、被说明、被引用。不能在某个文件里藏一个“只有作者知道”的能力。注册的意义不只是管理，更是为了让后续 AI 在上下文里知道自己到底有什么工具能用。

#### 1.7 所有工具统一由 Tool Runtime 调度
工具执行不是随便 import 一个函数就结束。工具执行应该有统一调度、统一输入输出、统一错误处理、统一日志。不同工具都可能走不同实现，但调度策略应该尽量统一。若工具层和业务层混在一起，后续很容易出现“改一个工具影响一串逻辑”的问题。

#### 1.8 所有配置集中管理
配置不要散落在多个 JSON、YAML、常量和代码里。集中管理不是为了好看，而是为了让后续能快速判断哪些是线上配置、哪些是本地测试、哪些是历史残留。特别是 sub2api、Gateway、Mother、Bridge、APK 端相关配置，必须明确来源与用途。

#### 1.9 所有日志统一输出
日志是回溯历史、排查问题、总结经验的关键。如果日志散落在多个地方，后续无法快速判断问题出在哪。统一日志不一定表示只有一个文件，而是要有统一格式、统一入口、统一归档与统一检索策略。

#### 1.10 所有新增模块必须先设计，再编码
先写大纲，再写接口，再决定目录，再开始实现。不能想到哪里写到哪里。这个要求不是官僚主义，而是为了防止结构先天畸形。尤其 MBclaw 这种多层系统，先设计才能避免后续返工。

#### 1.11 能借鉴现成品就必须借鉴
没有必要为了“自己重写”而重写。MCP、OpenHands、OpenClaw、LiteLLM、mem0、GraphRAG、Claude Code、Codex CLI、agentskills，能直接照搬的就直接照搬，不能直接照搬的再做本地化裁剪。关键不是“写得像不像原创”，而是“能不能把最合适的结构用到 MBclaw 上”。

#### 1.12 能减少代码量必须减少
MBclaw 不应该因为“看起来完整”而变得臃肿。能用一个清晰接口解决的，不要拆成一堆中间层；能复用的不要重复造轮子；能合并的不要硬拆。结构清晰比数量多更重要。

#### 1.13 不要只死磕某一种语言
语言是工具，不是信仰。Python 适合编排和原型，TypeScript 适合前后端结构，Go 适合服务和并发，Rust 适合安全和底层能力。哪个语言更适合当前问题，就用哪个语言。不要因为情绪绑定语言。

#### 1.14 禁止乱改重写
不要让你改 1.1.1，结果去动 1.1.2、1.1.3 的逻辑。改动必须局部、可追踪、可回退。MBclaw 的结构要可控，不允许一处小需求引起整仓库震荡。

#### 1.15 严禁动用批量工具做无脑改写
不允许为了省事批量替换造成全局误伤。尤其是在主线仓库、归档区、参考区之间，必须先理解再操作，尽量逐项核对。批量工具只能辅助搜索，不应该替代判断。

#### 1.16 完成代码的验收由两部分组成
第一层：在生产服务器或等价环境完成自测，确认功能可跑、没有明显 bug。第二层：由创建者确认功能是否通过。只有两层都满足，才算真正完成。没有用户确认的“自认为完成”，只能算内部阶段性结果。

## 第二篇：基础设施（Infrastructure）

### 第二章 统一模型调度中心（sub2api）

模型入口层的能力会优先整理到 sub2api 中；MBclaw 侧保留的是规则、边界、映射关系，不再并行维护第二套独立实现。凡是 LLM 相关请求，只要进入 MBclaw 的这条链路，就要先落到这套统一组织方式里，再决定最终由哪个 provider、哪个 key、哪个模型、哪个 fallback 路径来承接。

sub2api 这层的存在，决定了 MBclaw 是否能在多个模型、多个 key、多个 provider 之间稳定切换。没有这层统一组织，Mother 就会自己长出一堆调度逻辑；有了这层，Mother 只需要知道“我现在要什么能力”，不需要知道“我该去找谁”。

#### 2.1 Token 管理
Token 管理的主载体现在优先放在 sub2api，不再在 MBclaw 侧并行维护第二套同类结构。这里记录的不只是一个 key，而是完整记录 key / provider / model 的生命周期与治理状态。管理对象至少应包括：API Key、Base URL、Provider、Model、用途标签、备注、权重、优先级、是否可用、最近使用时间、失败次数、冷却时间、预算状态、所属分组与健康状态。

#### 2.2 Token 生命周期
Token 生命周期应当清晰：初始化 → 可用 → 检测 → 使用 → 失败 → 冷却 → 恢复 → 再加入。这个生命周期不是形式化描述，而是后续调度、诊断和管理后台展示的基础。没有生命周期，key 只是一串字符串；有了生命周期，key 才能成为可治理对象，并在 sub2api 的管理界面里被稳定追踪。

#### 2.3 自动健康检测
sub2api 承接的 sub2api 必须能识别失败、超时、速率限制、认证失效、模型下线等情况，并将其转化为可见状态。健康检测不是可选功能，而是调度系统的必备能力。失败次数、成功率、响应时间、最近调用时间、最近失败类型，都是后续评分、fallback、人工介入和后台告警的依据。

#### 2.4 Provider
Provider 不只是 OpenAI、Anthropic、Gemini、OpenRouter、Qwen、DeepSeek、Claude，也包括本地 Ollama、桥接服务、可能的中国区兼容服务。Provider 的职责，是把“模型从哪来”这件事抽象出来，让上层只关心能力，不关心接入形式。

#### 2.5 Model Registry
Model Registry 维护的不是名称列表，而是模型能力表。模型名称、上下文窗口、Vision、Function、Reasoning、速度、价格、限制、兼容性、推荐场景，都应该被记录。这样在选择模型时，不是凭感觉，而是根据任务类型和能力画像来匹配。

#### 2.6 Router
Router 负责模型评分、选择、降级、Fallback、Retry、负载均衡。Router 的最重要原则不是“选最强的”，而是“在预算、成功率、速度和能力之间做合理平衡”。Router 应该能识别：当前任务是推理、视觉、代码、搜索、短答还是批处理，从而决定优先级。

#### 2.7 Scheduler
Scheduler 不只是队列，也不是定时器，而是整个模型请求的调度器。Scheduler 需要考虑候选池、重试次数、失败冷却、预算约束、任务优先级、模型兼容性，最终决定发给哪个 provider。

#### 2.8 Statistics
统计不是事后报表，而是调度决策的依据。每日、每小时、每模型、每 key 的调用量、成功率、耗时、失败类型、成本估算，都应该进入统计体系。统计体系做得好，后续才能回答“哪个 key 适合什么任务”。

#### 2.9 OpenAI Compatible API
sub2api 最重要的对外接口之一，是 OpenAI Compatible API。原因不是因为 OpenAI 最好，而是因为这个协议形状最容易兼容上层调用。只要保持兼容格式，上层就能用统一方式调用多 provider，不需要每换一个 provider 就改一套代码。

#### 2.10 管理后台接口
sub2api 的管理后台应支持：查看 token、查看 provider、测试 key、查看统计、查看冷却、调整优先级、禁用/启用 key。管理后台的作用，是让 sub2api 不只是“自动跑”，还要“可管、可看、可控”。

### 第三章 独立控制面板（sub2api Admin）

sub2api Admin 是专门管理 sub2api 的控制台。它的价值在于把模型入口的运维、健康检测、Key 维护、Provider 管理、统计查询和异常处理从 Mother 的主业务里分离出去。这样一来，Mother 不用处理复杂的 Key 管理界面，也不用把管理逻辑和业务逻辑混在一起。

#### 3.1 Dashboard
Dashboard 需要展示：总 key 数、可用 key 数、冷却 key 数、失败 key 数、最近调用量、今日耗用、各 provider 状态、最近异常。Dashboard 的目标是“一眼看懂当前模型入口是否健康”。

#### 3.2 Provider
Provider 页面负责录入和维护 provider 相关信息，包括 base_url、api_key、模型列表、兼容接口、优先级、是否启用。

#### 3.3 Model
Model 页面负责整理模型能力与使用建议，避免上层调用时对模型一无所知。

#### 3.4 Key
Key 页面负责新增、删除、禁用、测试、批量检测、分组、标签、备注等管理动作。

#### 3.5 日志
日志页面应展示成功、失败、429、401、超时、熔断、冷却、重试等关键事件。

#### 3.6 API
管理 API 用于前端页面、脚本、自动化任务调用，不仅用于显示，也用于管理动作。

#### 3.7 用户
如果未来要做多人或授权，用户维度才进入 sub2api Admin 的组织结构；当前阶段可以先保留接口位，但不要先做复杂权限系统。

#### 3.8 权限
sub2api 的权限体系应尽量轻，重点是保护 key 与统计，不要把它做成完整企业 IAM。

## 第三篇：MBclaw 总控制中心（Control Center）

MBclaw 总控制中心不是 sub2api，也不是 Mother。它是整个 MBclaw 体系未来统一后台入口的候选位置，负责服务总览、日志中心、配置中心、用户中心、权限系统、更新中心、插件中心、Marketplace 等统一管理能力。当前阶段，总控制中心更多是“应该长成什么样”的设计框架，而不是已经全部落地的产品。

#### 4.1 Dashboard
总览要能显示：Mother、Gateway、sub2api、Android、Windows、Linux、Web、API、Memory、Worker、设备、用户、在线状态、错误状态、最近活动。

#### 4.2 服务管理
应当能够看到 systemd、Docker、Compose、Cluster、脚本、守护进程等服务运行状态。即使当前不启用某些部署方式，也应预留界面位。

#### 4.3 日志中心
所有日志统一检索，按服务、时间、级别、任务、用户分类，方便未来排错与回溯。

#### 4.4 配置中心
统一管理环境变量、配置文件、开关、密钥引用、部署参数。

#### 4.5 用户中心
用户中心应服务于未来账号与设备体系，不是当前阶段重点，但应预留。

#### 4.6 权限系统
权限系统要区分管理者、设备、桥接服务、子代理、未来扩展身份。

#### 4.7 更新中心
管理版本、迁移、备份、回滚、灰度，不把更新散落在各个仓库里。

#### 4.8 插件中心
为后续插件化、能力包、集成项提供入口。

#### 4.9 Marketplace
Marketplace 是生态层接口的预留，不是当前阶段主攻方向。

## 第四篇：Mother（MBOS）

Mother 是整个 MBclaw 的核心。它不是一个简单的聊天接口，而是把消息接入、上下文、记忆、规划、执行、工具、反馈、状态管理放在一起的运行中枢。Mother 的定位是“能持续工作的大脑”，而不是“只会回答问题的模型外壳”。

### 第四章 Gateway

Gateway 是唯一入口。任何平台都必须先经 Gateway，再进入 Mother 的标准化流程。Gateway 的职责是接收、验证、标准化、路由、回复，而不是思考、记忆、执行。

#### 5.1 Adapter
Gateway 必须支持多种 Adapter：QQ、微信、飞书、Telegram、Discord、Web、API、APP、CLI、Voice、Vision 等。当前已存在的只是其中一部分，未来可以扩展，但核心原则不变：平台差异在入口层解决。

#### 5.2 Normalize
所有消息在进入 Mother 前，必须统一成 StandardMessage。统一消息格式是后续任务拆解、记忆、追踪、统计的基础。

#### 5.3 Session
Gateway 要负责把平台消息映射到统一 Session，从而把来自不同渠道的输入归到稳定的会话结构里。

#### 5.4 Identity
身份识别不是为了复杂权限，而是为了知道“是谁在说话、从哪来、对应哪条会话”。

#### 5.5 Upload
附件、图片、视频、语音等上传能力，必须在 Gateway 层统一处理。

#### 5.6 Reply
不同平台的回复格式转换，也应在 Gateway 层处理。

### 第五章 Mother Runtime

真正的大脑是 Runtime，而不是任何单点模型。Runtime 负责接收标准消息，维护工作记忆，调度上下文，选择工具，循环执行，生成结果，再把结果交给 Gateway 输出。

#### 6.1 Governor
Governor 是系统宪法层，负责高层约束、策略、边界、风险控制。

#### 6.2 Planner
Planner 负责任务拆解，把复杂目标拆成可执行步骤。

#### 6.3 Scheduler
Scheduler 负责能力调度，不只调模型，也调工具和执行顺序。

#### 6.4 Context Engine
Context Engine 只负责当前会话上下文组织，不负责长期存储。

#### 6.5 Working Memory
Working Memory 负责当前会话状态，短期保留。

#### 6.6 Long Memory
Long Memory 负责长期知识、经验、用户习惯、项目结论的沉淀。

#### 6.7 Capability Registry
Capability Registry 负责工具和技能的注册、摘要、索引与发现。

#### 6.8 Skill Runtime
Skill Runtime 负责执行技能，不要让技能散落在不同调用链中。

#### 6.9 Observation
Observation 负责系统观察、状态变化、行为反馈、失败记录。

#### 6.10 Evolution

## 第八篇：施工规范（Development Specification）

### 第一章 总原则

施工规范的作用，是防止项目在未来扩展时走样、失控或越改越乱。所有后续模块都必须遵守这些规范。

#### 1.1 新模块必须先设计后编码
任何新功能、新目录、新接口都应该先有文档和结构，再写实现。

#### 1.2 接口先行
设计阶段先预留标准接口文件，即使暂不实现，也要先把接口位置和用途说明放好。

#### 1.3 不跨层访问内部实现
跨模块通信必须通过标准接口，不允许随手 import 内部类直接操作。

#### 1.4 先保主线，再做优化
如果冲突，优先保证 MBclaw 能启动、能运行、能恢复，再谈重构美观。

#### 1.5 小步提交思想
每次施工尽量只处理一个主题，不要把多个主题混在一起。

### 第二章 接口预留规范

每个模块至少包含：

```
interfaces/
├── api.py
├── service.py
├── events.py
├── models.py
├── errors.py
└── README.md
```

#### 2.1 api.py
对外 API，负责稳定调用入口。

#### 2.2 service.py
内部业务接口，负责业务实现。

#### 2.3 events.py
事件定义，负责模块通信约束。

#### 2.4 models.py
数据模型，负责结构说明。

#### 2.5 errors.py
错误定义，负责异常收敛。

#### 2.6 README.md
必须说明接口用途、调用方、被调用方、输入、输出、时机、生命周期、兼容性、扩展点。

### 第三章 统一调用规范

统一调用链：

Caller → Interface(API) → Service → Runtime → Result

禁止跨模块直接访问内部实现，不允许跳过中间层直接碰底层逻辑。

### 第四章 事件规范

事件统一为：

```
Event {
  id
  source
  target
  type
  timestamp
  payload
  metadata
}
```

所有跨模块通信优先通过事件总线完成，避免散乱调用。

### 第五章 版本规范

每个接口都必须维护：Version、Deprecated、ChangeLog。后续升级必须考虑向后兼容。

## 第十一篇：各章施工细化说明（扩写版）

### 第十一章 sub2api 施工细化

sub2api 相关能力现在以 sub2api 作为整理主载体。sub2api 不是一个“拿 key 发请求”的简单工具，而是 MBclaw 的能力中枢之一；这套能力要优先映射到 sub2api 的 backend、frontend、tools、skills 与 docs 结构中，MBclaw 侧只保留原则、边界和引用，不再并行维护第二套同类实现。sub2api 如果不细化，Mother 就会被迫自己处理路由、冷却、失败恢复、统计、预算和多 provider 差异，最终变成一坨难以维护的逻辑。

#### 11.1 sub2api 的职责边界再定义
模型入口概念上仍代表“模型入口治理”，但当前整理主载体是 sub2api；MBclaw 侧不再并行维护第二套同类入口。它可以知道一个请求是短答、长文、推理、视觉、代码、搜索，但它不应该知道“用户为什么要问这个问题”。原因很简单：目标理解属于 Mother，资源治理属于 sub2api。两个层次不能混在一起。

sub2api 应该做的事情包括：
- 接收统一请求描述。
- 根据任务类型筛选候选 provider。
- 根据 key 状态、失败次数、冷却状态、预算状态、模型能力，构建可用候选列表。
- 尝试 fallback。
- 在失败时更新统计。
- 在成功时记录耗用与最后使用时间。
- 在管理页面中可视化状态。

sub2api 不应该做的事情包括：
- 决定用户最终要表达什么。
- 决定消息是否需要改写。
- 决定工具如何执行。
- 决定对话记忆怎么拼。
- 决定 Gateway 怎样接消息。

#### 11.2 sub2api 的理想数据对象
sub2api 的 key、provider、model 三者不能混成一个模糊对象。它们是不同层级的东西。

- **Key**：代表一个具体可用的认证凭据或调用凭据。
- **Provider**：代表一个上游服务商或中转来源。
- **Model**：代表具体能力单位，例如推理、视觉、文本、代码、embedding。

如果把这三层混在一起，后续就会出现“一个 key 被误当成一个 model”、“一个 provider 被误当成一条 API 路径”的情况，维护成本会很高。

#### 11.3 冷却、重试、降级、fallback 的顺序
sub2api 在面对失败时，处理顺序应当固定：
1. 先判断是否短暂失败。
2. 再判断是否进入冷却。
3. 再判断是否触发 fallback。
4. 再判断是否切换 provider。
5. 最后再判断是否返回失败。

不能一上来就把所有错误都当成“key 坏了”。429、401、5xx、连接超时、网络波动、模型下线，代表的含义完全不同。把所有异常统一当成“失败”会导致冷却过度或误杀。

#### 11.4 sub2api 的管理页面应该展示什么
sub2api Admin 不只是看 key 列表，而是要让人一眼看懂整个模型调度系统是否健康。页面至少要有：
- 今日请求量
- 今日消耗量
- 当前可用 key 数
- 冷却 key 数
- 连续失败 key 数
- 成功率最低 provider
- 最近失败类型
- 最常用模型
- 最近恢复的 key
- 需人工介入的异常

#### 11.5 sub2api 的验收标准
sub2api 的验收不是“能发请求”这么简单，而是：
- 能正确识别可用 key。
- 能正确识别失败 key。
- 能在失败后进入冷却。
- 能在多个 provider 中切换。
- 能正确记录统计。
- 能正确被管理后台查看。
- 能正确兼容 MBclaw 当前主线调用方式。

### 第十二章 Gateway 施工细化

Gateway 是 MBclaw 的第一道门。它的工作不是思考，而是把来自不同平台的输入统一成标准结构。Gateway 要做的是把“杂乱的外界输入”变成“内部可处理对象”。

#### 12.1 Gateway 的输入与输出
Gateway 输入是平台原始消息。输出是 StandardMessage。这个转换过程必须稳定、可追踪、可扩展。

输入常常是：
- QQ 文本消息
- 微信文本消息
- 群消息
- 语音转文字
- 图片识别后的文本
- Web 聊天框内容
- CLI 命令文本

输出应统一为：
- channel
- user_id
- content
- content_type
- timestamp
- metadata
- attachment_refs

#### 12.2 Gateway 的会话策略
一个通道可能有多个用户，一个用户也可能同时在多个通道发消息。Gateway 的会话策略必须清晰，否则 Memory 和 Runtime 都会混乱。至少要明确：
- 当前会话如何生成。
- 多平台是否共享会话。
- 是否允许同一用户跨平台聚合。
- 群聊与私聊是否分开。
- 未来是否支持设备级会话。

#### 12.3 Gateway 的鉴权与身份
Gateway 负责最初级身份识别，但不负责重权限判断。它只需要知道消息是不是来自合法通道、是不是应该进入系统、是不是需要附带额外元数据。

#### 12.4 Gateway 的验收标准
Gateway 的验收标准包括：
- 能接收所有已支持通道消息。
- 能稳定转成统一格式。
- 能正确映射会话。
- 能正确回传回复。
- 不把平台逻辑混入母体。

### 第十三章 Mother Runtime 施工细化

Mother Runtime 是 MBclaw 的中枢。它不是“调用 LLM 然后回一句话”那么简单，而是要完成一个完整的工作循环：接收消息、建立上下文、查询记忆、决定工具、调度模型、执行动作、总结结果、回传消息、记录经验。

#### 13.1 Runtime 的基本循环
一个标准 Runtime 回合至少包括：
1. 接收标准消息。
2. 解析会话与用户。
3. 拉取工作记忆。
4. 拉取长期记忆。
5. 拉取项目状态。
6. 拉取可用工具。
7. 组装上下文。
8. 选择模型。
9. 执行输出。
10. 判断是否需要继续工具调用。
11. 判断是否需要继续下一轮。
12. 保存结果。
13. 更新记忆。
14. 返回 Gateway。

#### 13.2 Runtime 的循环边界
Runtime 不能无限循环。需要明确：
- 最大轮次。
- 单次工具调用超时。
- 模型失败重试上限。
- 无结果终止条件。
- 用户新消息插队规则。

#### 13.3 Runtime 的上下文注入顺序
上下文注入不是随便堆积，而是要有顺序。一般建议：
1. 系统宪法 / Governor。
2. 当前会话目标。
3. 项目规则。
4. 长期记忆摘要。
5. 当前用户消息。
6. 工具列表。
7. 当前状态。
8. 历史关键结果。

#### 13.4 Runtime 的工具原则
工具调用应该可见、可追踪、可回放。每次工具调用都应有：工具名、输入、输出、耗时、结果、失败原因。

#### 13.5 Runtime 的验收标准
Runtime 的验收至少要满足：
- 能从 Gateway 接收统一消息。
- 能构建上下文。
- 能调用模型。
- 能调用工具。
- 能存储和更新经验。
- 能正确返回结果。

### 第十四章 Memory 施工细化

Memory 不是一个简单存档，而是整个系统未来能否“越用越懂用户”的关键。

#### 14.1 为什么要分九层
分层不是为了文档好看，而是为了让不同类型的信息有不同生命周期。会话摘要不该和用户画像混在一起，决策记录不该和临时输出混在一起，经验教训不该和知识条目混在一起。

#### 14.2 Conversation 层的作用
Conversation 层负责把一次对话提炼成可以回看的摘要，保存关键主题、关键结论、关键冲突点。这个层级最适合回答“刚才聊了什么”。

#### 14.3 Project 层的作用
Project 层负责把一个长期项目的目标、规则、阶段、完成状态、风险点、验收标准固定下来。这个层级最适合回答“这个项目现在做到哪了”。

#### 14.4 Decision 层的作用
Decision 层负责保存“当时为什么这么选”。这比结果本身更重要，因为它能帮助未来判断哪些路曾经尝试过、哪些路行不通、哪些路值得复用。

#### 14.5 Knowledge 层的作用
Knowledge 层负责沉淀长期有效的事实、方法、结论、规则。它不是临时结论，而是能被复用的知识资产。

#### 14.6 Experience 层的作用
Experience 层负责保存成功、失败、尝试、教训。它特别适合帮助未来任务避免重复踩坑。

#### 14.7 Capability 层的作用
Capability 层负责记录工具和技能在什么场景下更合适，让系统以后知道“什么工具更顺手”。

#### 14.8 Observation 层的作用
Observation 层负责记录系统和设备状态变化，例如服务在线、任务中断、设备异常、用户行为变化。

#### 14.9 User 层的作用
User 层负责保存用户偏好、语言、设备、工作风格、常用工具、常见指令。

#### 14.10 Evolution 层的作用
Evolution 层负责保存系统本身的改进史，方便审计和回顾。

#### 14.11 Memory 的验收标准
Memory 的验收不只是“能存”，而是：
- 能分类存。
- 能正确查。
- 能正确更新。
- 能给上下文真正用上。
- 能避免重复记忆污染。

## 第十五篇：重点仓库逐项说明（加厚版）

这一篇的目的，是把当前最重要的参考仓库、主线仓库和能力仓库，按照“为什么要看、该看什么、该保留什么、该淘汰什么”的方式写清楚。这样后续接手的人不用再自己猜测优先级，也不需要每次重新搜索历史结论。

### 第十七章 mbclaw-mother

mbclaw-mother 是当前主线中的主线，是 MBclaw 现实可运行体系最核心的仓库。它不是单纯的母体原型，而是已经包含 Mother、Gateway、sub2api、Control Panel、Memory、工具、桥接、健康检查和统计等真实运行组件的系统仓库。对这个仓库的理解，决定了后续整个 MBclawZZ 是朝着“主线收敛”还是“继续散开”走。

#### 17.1 为什么它是 P0
因为它直接回答了三个最关键的问题：
- MBclaw 现在到底长什么样？
- Mother、Gateway、sub2api 的边界到底在哪里？
- 当前系统里哪些东西是真正在跑的，哪些只是概念或残留？

如果这三个问题不先回答，后面所有整理都会失焦。

#### 17.2 需要重点看的内容
- Mother Runtime 的调用链。
- Gateway 的消息处理。
- sub2api 的 key 管理与 fallback。
- Control Panel 的统计和管理。
- Memory 的数据结构和检索方式。
- 与真实服务运行相关的配置和路径。

#### 17.3 可保留内容
- 真实运行链路。
- 已经验证过的结构。
- 可直接指导 MBclawZZ 施工的结论。
- 已确定的边界说明。

#### 17.4 可淘汰内容
- 过时备份。
- 重复版本。
- 早期原型壳。
- 不再使用的临时文件。

#### 17.5 对 MBclawZZ 的价值
它是“主线来源”，不是普通参考。后续所有总纲、对标、归档、总结，都应该围绕它展开。

### 第十八章 OpenClaw

OpenClaw 是最值得研究的平台化参考之一。它的价值不是“文件很多”，而是它把 channels、context-engine、skills、plugins、agent 组织成了一个完整的平台结构。

#### 18.1 为什么看 OpenClaw
因为它展示了一个更成熟的 Agent OS 轮廓：
- 多渠道接入
- 上下文引擎独立
- 技能系统清晰
- 插件注册体系完整
- 架构层次丰富

#### 18.2 MBclaw 应该学什么
- 渠道插件化。
- context engine 结构。
- skills 的声明式组织。
- 用户↔渠道绑定。
- 任务/会话/线程的分层思路。

#### 18.3 不要直接照搬什么
- 巨大的 TypeScript 全栈结构。
- 过度复杂的渠道体系。
- 与 MBclaw 当前规模不匹配的重设计。

#### 18.4 结论
OpenClaw 适合做“架构天花板”参考，不适合直接复制代码。

### 第十九章 OpenHands

OpenHands 更适合看 runtime 分层、SDK/Server 分离、DI、integration 层、工具执行与开发者工作流。它不是 MBclaw 的直接模板，但它提供了一个“把 Agent 做成可执行系统”的成熟参考。

#### 19.1 应该看什么
- runtime 如何组织。
- SDK 与 server 的边界。
- integrations 的组织。
- 工具与执行的分层。
- 会话和任务如何推进。

#### 19.2 MBclaw 能借的地方
- 依赖注入。
- 运行时分层。
- 工具系统组织。
- 外部集成接口的规范化。

#### 19.3 不适合直接引入的地方
- 沙箱/容器过度依赖。
- 面向开发者的产品目标。
- 与 MBclaw 当前个人助理定位不一致的复杂机制。

### 第二十章 LiteLLM

LiteLLM 是 sub2api 最重要的外部对照之一。它的价值不是模型种类多，而是它在 cooldown、retry、fallback、rate limit、deployment、routing 方面形成了成熟思路。

#### 20.1 该借什么
- 冷却策略。
- 重试策略。
- Fallback 链。
- Rate limit 判断。
- 失败恢复思路。

#### 20.2 不该借什么
- 企业级预算系统。
- 过于复杂的 router strategy。
- 大量与 MBclaw 无关的 guardrails 和多租户逻辑。

#### 20.3 与 sub2api 的关系
sub2api 应优先判断哪些能力可以直接复用或照搬，再把不适合原样搬入的部分做收口裁剪，目标是形成更轻量、更可维护的调度器。

### 第二十一章 mem0

mem0 适合看“记忆如何被更新”，而不是“记忆如何做得很大”。MBclaw 当前阶段更需要的是清晰可用的记忆层，而不是一上来就上复杂向量 DB 和多层云服务。

#### 21.1 该借什么
- 记忆增量更新。
- 用户隔离。
- 记忆分类。
- add / update / delete 语义。

#### 21.2 不该直接引入什么
- 外部向量 DB 强依赖。
- 不必要的成本型检索层。
- 过早复杂化的知识存储方式。

#### 21.3 当前阶段的定位
作为中长期记忆设计对照，而不是直接施工目标。

### 第二十二章 GraphRAG

GraphRAG 是“知识关系”方向的远期参考，适合看层级摘要、关系抽取、知识连接。当前阶段它不是必须品，但可以作为未来知识记忆增强的灵感来源。

### 第二十三章 MCP

MCP 的价值在于工具定义、schema、发现机制、协议接口形状。MBclaw 不一定要引入 MCP 的远程协议，但它的工具定义方式很值得参考。

### 第二十四章 Claude-Code 与 Codex-CLI

这类仓库适合看终端代理如何组织命令、计划、上下文和执行。它们不一定适合直接变成 MBclaw 的运行结构，但可以帮助我们理解“命令式 AI 助手”的边界。

### 第二十五章 agentskills

agentskills 适合看技能如何写得简洁、可发现、可组织。它对 MBclaw 的价值在于：技能不应该只有执行逻辑，还应该有足够短的说明、标签和用途描述。

### 第二十六章 FreeLLMAPI 与 miclaw_api_bridge

这两个方向更像桥接/中转/兼容层参考。它们可以帮助理解“如何在模型入口、设备桥接、API 适配之间建立转换层”，但不应该成为 MBclaw 主线的重心。
